很多人把 OpenClaw 理解成“一个能聊天、能调工具的助手外壳”。这理解不能说错，但有点小了。

它更像一层**个人资讯操作系统**：上面接输入源，下面接分发渠道，中间再塞上记忆、自动化、浏览器、节点、定时任务、消息路由。你要它查新闻、看网页、读 RSS、抓社交平台、整理日报、推送到 Telegram，甚至顺手沉淀成博客文章，它都不是不能干；问题从来不是“能不能”，而是**该怎么编排，在哪一层做，安全边界怎么卡住**。

这篇文章只研究一件事：**OpenClaw 如何获取资讯，以及如何展示资讯。**

重点不是花哨 demo，而是把“抓取 → 处理 → 展示 → 推送”这条链路讲透，顺便给出一套适合 lemon / 可乐 当前系统的落地方案。

---

## 一、先下结论：OpenClaw 适合做什么，不适合做什么

### 适合做的

OpenClaw 很适合当下面这些角色：

- **统一入口**：把网页、搜索、RSS、GitHub、社交平台、视频平台、消息渠道收进同一个工作流里
- **编排层**：把搜索、抓取、摘要、去重、归档、发布串起来
- **个性化分发器**：同一份资讯，既可以发 Telegram，也可以沉淀成博客长文、日报、专题页
- **跨设备协调器**：服务端跑定时任务，必要时调用本地浏览器或节点去处理需要登录态、可视化页面、文件上传的任务
- **带记忆的内容助手**：知道你长期关注什么、讨厌什么、哪些信息值得长期保存

### 不适合做的

但它也不是万能吸尘器。

OpenClaw 不适合直接承担：

- **高频、大规模、工业级爬虫集群**：那是 Scrapy、Airflow、Kafka 那类系统的地盘
- **极重的数据清洗与搜索基础设施**：比如海量全文索引、复杂向量召回、超大规模 ETL
- **需要长期稳定绕过反爬的平台采集**：尤其强登录态、强风控、强验证码的平台
- **完全无人监管的对外发布**：尤其是自动发帖、自动外呼、自动批量传播，翻车概率比你以为的大

一句话：

> **OpenClaw 最擅长的是“中等规模、以个人/小团队为中心、强调多源整合和智能编排”的资讯系统。**

它不是新闻门户后端，但它非常适合做一个属于你自己的“早报姬 + 研究台 + 发布台”。

## 二、资讯获取层：OpenClaw 到底能从哪儿拿信息

如果把资讯系统拆开，第一层永远是输入源。OpenClaw 的强项在于：它不是只给你一个搜索框，而是给你**多种不同精度和成本的获取方式**。

按从“轻”到“重”来排，大致可以分成五类。

---

## 三、第一类：`web_search` —— 快速找线索，适合做入口层

`web_search` 是 OpenClaw 的标准网页搜索入口。当前文档里推荐的搜索提供方是 **Brave Search**；它负责给你返回结构化搜索结果，而不是直接把整个网页塞给模型。

### 它适合干什么

最适合干三类事：

- **发现候选来源**：今天有哪些新闻、有哪些新文章、哪些官方博客更新了
- **做主题扫描**：比如“OpenAI 新模型”“浏览器代理安全”“Agent 框架更新”
- **辅助选题**：先拉一个主题的结果面，再决定哪些值得深读

### 优点

- **快**：查入口、找线索的效率很高
- **结构化**：返回标题、链接、摘要，比让浏览器一页页翻省很多 token
- **可过滤**：国家、语言、时间范围、新鲜度都能收窄
- **适合定期 briefing**：比如“最近一周 AI 浏览器代理的更新”

### 缺点

- **只告诉你哪里可能有料，不负责把料读透**
- **依赖搜索提供商覆盖面与排序逻辑**
- **强平台内内容不一定好抓**：例如部分封闭社交内容、需要登录态的页面
- **有配额与条款约束**：尤其是商用、自动化、缓存和再分发场景

### 适用场景

我建议把 `web_search` 放在资讯链路的**第一跳**：

1. 先查一轮主题
2. 拿到候选 URL
3. 再把候选 URL 交给 `web_fetch`、浏览器、Agent Reach 工具链做深读

它像侦察兵，不像搬运工。

---

## 四、第二类：`web_fetch` —— 直接读网页正文，适合正文抽取

如果 `web_search` 负责“找到哪里有东西”，那 `web_fetch` 负责“把网页正文抽出来”。

它的价值非常直接：**把 HTML 页面转成适合模型处理的 markdown/text**。对于新闻、博客、文档、公告页，这玩意儿非常顺手。

### 它适合干什么

- 读新闻正文
- 读官方博客更新
- 读产品更新日志
- 读文档页、帮助中心、公告页
- 给后续摘要、分类、比较做干净输入

### 优点

- **轻量**：不需要完整浏览器渲染
- **成本低**：比浏览器便宜、稳定、快
- **非常适合内容型网页**：博客、文档、公告、新闻站正文
- **天然适合摘要链路**：提取后直接喂给模型做总结、归纳、比对

### 缺点

- **对动态站点、前端渲染重站点不总是好使**
- **拿不到复杂交互后的内容**
- **登录态页面通常不行**
- **页面结构太脏时可能抽取不完整**

### 适用场景

如果一个链接是“公开可读、正文为主、交互不复杂”，优先用它。

这点很重要：

> **能用 `web_fetch` 解决，就别急着上浏览器。**

因为浏览器虽然猛，但也重、慢、容易踩反爬。你不需要每次为了看一篇博客都抬出坦克。

---

## 五、第三类：`browser` —— 重武器，处理动态页面、登录态和复杂交互

OpenClaw 的 `browser` 工具不是装饰品，它是完整的浏览器控制能力：开页面、快照、点击、输入、上传、截图、导 PDF、处理对话框，甚至能走 Chrome 扩展接管已有标签页。

这意味着什么？

意味着 OpenClaw 不只是“读网页”，而是可以**操作网页**。

### 它适合干什么

- 动态网站内容抓取
- 需要点击“展开全文”“加载更多”的页面
- 需要登录态的后台、控制台、社区网站
- 抓一些 `web_fetch` 抽不出来的内容
- 截图式留档、可视化验证
- 从复杂页面里抽结构化信息

### 优点

- **能力最完整**：几乎等于远程操控浏览器
- **能处理动态渲染和前端交互**
- **可在已有 Chrome 标签页上接管**，适合利用本地登录态
- **可配合 nodes** 在别的设备上操控本地浏览器

### 缺点

- **重**：资源消耗高，速度慢于搜索和抓取工具
- **脆**：网页改版、元素变化、反爬策略都会导致流程不稳
- **安全风险高**：因为它可能接触登录态、cookie、会话页面
- **不适合当默认手段**

### 适用场景

浏览器应该作为**兜底层和特攻层**：

- 公开网页正文 → `web_fetch`
- 搜索发现 → `web_search`
- 动态复杂站点 / 登录态 / 必须交互 → `browser`

也就是说，浏览器该用，但别滥用。

否则系统会变成一个很累、很贵、还动不动被网站抽耳光的自动点点点机器人。

---

## 六、第四类：Agent Reach 工具链 —— 把“泛网页”扩成“平台级资讯接入”

如果说 `web_search` / `web_fetch` / `browser` 解决的是**网页层**，那 Agent Reach 解决的是**平台层**。

这套能力的意义很大：它把 OpenClaw 的资讯输入源，从“能看网页”扩成了“能接更多平台原生接口或专用抓取通道”。

按目前技能说明，它覆盖或可接入的渠道包括：

- Twitter / X
- Reddit
- YouTube
- GitHub
- Bilibili
- 小红书
- 抖音
- LinkedIn
- Boss 直聘
- RSS
- 任意网页（Jina Reader / Exa / mcporter 生态）

这就是典型的**资讯采集扩展层**。

### 它为什么重要

因为现实世界的资讯并不都老老实实地待在公开 HTML 页面里。

很多一手信息分布在：

- GitHub issue / release / PR
- X 上的作者动态
- YouTube / B站视频及字幕
- Reddit 社区讨论
- RSS Feed
- 平台专用页面与内容流

如果只有 `web_search` 和 `web_fetch`，你能做的资讯系统仍然偏“网页新闻聚合器”；
而有了 Agent Reach，OpenClaw 才更像一个**真正的跨平台研究助手**。

---

## 七、Agent Reach 各类渠道的价值与代价

### 1. GitHub：最适合做技术资讯源

GitHub 资讯天然适合 OpenClaw。

因为它的信息结构清晰、更新节奏稳定、内容可追踪：

- release
- issue
- PR
- commit
- discussion
- README / docs 变化

#### 优点

- 权威、一手、可追溯
- 结构化程度高
- 特别适合技术项目监控、竞品跟踪、开源情报

#### 缺点

- 对非技术用户不够友好
- 信息噪音也不少，需要筛选层

#### 最适合的用法

- 监控指定仓库的 releases / issues / PR
- 生成“本周项目动态”
- 为博客专题页提供原始素材

---

### 2. RSS：最稳、最干净、最被低估的资讯源

RSS 这玩意儿老，但真香。

它的优点不是“先进”，而是**稳定、明确、低噪音**。

#### 优点

- 标准化程度高
- 非常适合定时抓取
- 变动成本低
- 适合博客、媒体、公告源、更新日志
- 对隐私相对友好

#### 缺点

- 不是所有平台都有
- 有些 feed 内容只有摘要，没有全文
- 平台化社交内容覆盖差

#### 最适合的用法

- 当做“基础信号层”
- 跟踪高质量来源：产品博客、技术博客、研究机构、媒体、播客更新
- 做稳定日报，不容易炸

如果你想要一个**不靠玄学、能长期跑的资讯系统**，RSS 应该是骨架之一。

---

### 3. X / Reddit / 社交平台：一手快，但风控和噪音都大

社交平台的价值很明显：**快、原始、未被媒体加工**。

但它们的问题也很明显：**脏、乱、反爬强、上下文碎片化**。

#### 优点

- 一手消息早
- 能看到讨论温度、争议点、社区反馈
- 很适合主题雷达和人物追踪

#### 缺点

- 反爬、登录态、cookie、代理问题多
- 数据稳定性差
- 噪音高，容易被情绪和流量算法污染
- 法律与平台条款风险更敏感

#### 最适合的用法

- 作为“信号发现层”，而不是最终事实层
- 拿来发现选题、捕捉舆情、追踪关键人物
- 发现后再回查官方博客、GitHub、公告、视频、文档做交叉验证

社交平台非常像路边八卦摊：**有时第一时间知道事，但你不能直接拿来写结论。**

---

### 4. YouTube / B站 / 视频平台：获取“口头信息”与演示细节

很多真正有价值的信息，并不是文章，而是：

- 发布会视频
- 开发者直播
- 访谈
- 教程演示
- 作者口播更新

视频平台的价值在于能拿到**文章里没有的语气、细节、现场感和演示过程**。

#### 优点

- 可获得字幕、标题、描述、元信息
- 适合做内容摘要与观点提炼
- 对“怎么做”“演示了什么”特别有帮助

#### 缺点

- 成本高于普通网页
- 字幕质量不稳定
- 平台反爬与地区限制问题常见
- 纯视频内容很长，容易浪费处理预算

#### 最适合的用法

- 只抓你明确关心的频道 / 视频
- 先抓 metadata，再决定要不要拉字幕
- 把视频当成深度素材，而不是广谱输入源

---

## 八、第五类：消息渠道与用户输入，本身也是资讯源

这是很多人会忽略的一点：

> **OpenClaw 的聊天渠道本身，也是一种资讯输入层。**

比如 Telegram、Discord、Slack、Signal、WhatsApp 等，不只是“输出口”，也可以是：

- 用户转发来的链接
- 群里丢来的新闻线索
- bot 收到的提问与主题指令
- 某个频道的消息事件流

这意味着 OpenClaw 可以把“外部链接 + 用户意图”组合起来。

例如：

- 用户转一篇文章 → 让 OpenClaw 做摘要 + 判断值不值得归档
- 用户发一句“给我盯着这个方向” → 加入长期监控主题
- 某群里不断出现同一个项目名 → 自动触发专题跟踪

这类输入不是新闻源本身，但它是**很好的触发器**。

## 九、不同获取方式的取舍：别只问能不能，要问值不值

下面给一个更工程化的判断表。

### `web_search`

- **定位**：入口发现
- **强项**：快、广、便于主题扫描
- **弱项**：不读全文、不保上下文
- **推荐度**：高
- **默认优先级**：高

### `web_fetch`

- **定位**：正文抓取
- **强项**：轻、稳、便宜、适合公开网页
- **弱项**：对动态站和登录态无能为力
- **推荐度**：很高
- **默认优先级**：最高

### `browser`

- **定位**：交互与复杂页面处理
- **强项**：能点击、能登录、能截图、能处理动态站点
- **弱项**：重、慢、脆、风险高
- **推荐度**：中高
- **默认优先级**：兜底

### Agent Reach / 平台工具链

- **定位**：平台级资讯接入
- **强项**：覆盖 GitHub、社交、视频、RSS 等专用源
- **弱项**：安装、维护、认证、代理、cookie 都更复杂
- **推荐度**：很高
- **默认优先级**：按平台需要启用

### 用户消息 / 群聊输入

- **定位**：人工触发、线索注入
- **强项**：与用户意图天然绑定
- **弱项**：不稳定、不可规模化
- **推荐度**：高
- **默认优先级**：作为触发器保留

结论其实很朴素：

> **默认组合应该是：`web_search` 找入口，`web_fetch` 读正文，Agent Reach 负责平台源，`browser` 负责兜底。**

这四层配好了，绝大多数个人资讯系统就能跑得很像样。

## 十、隐私、安全与合规：资讯系统最容易翻车的地方

资讯采集不是只要技术通了就万事大吉。真正容易出事的，是安全边界。

### 1. 搜索和网页抓取的风险

风险相对较低，但仍要注意：

- 不要把敏感 token 拼进 URL
- 不要把抓取到的外部页面内容当成可信指令
- 不要让模型盲信外部网页里的“系统提示”“请继续执行”这类垃圾注入

对 OpenClaw 来说，外部页面应一律视为：**不可信内容源，不可信指令源**。

### 2. 浏览器与登录态的风险

这是高危区。

因为浏览器工具可能接触：

- 已登录后台
- cookie / session
- 私有页面
- 个人账号内容
- 上传或发帖按钮

所以必须有两条铁律：

- **读和写分离**：采集资讯尽量使用只读方式，避免碰任何提交、发布、转发动作
- **登录态最小化**：只有确实需要的平台才走浏览器接管或本地登录态

### 3. Agent Reach / Cookie / 代理的风险

这一层是信息获取能力最强、同时也是安全最容易炸的地方。

问题包括：

- cookie 泄露
- 平台封禁
- 代理服务不可信
- 第三方 MCP / CLI 工具带来的供应链风险
- 不同平台条款对自动化抓取的限制

所以平台型接入不能一股脑全开。正确做法是：

- **按需启用，不做全家桶炫技**
- **优先官方、公开、可审计的接口与工具链**
- **登录态平台使用专用账号或受限账号**
- **在本地与服务端之间分清边界**

### 4. 输出侧的风险

资讯系统最后会走向“发布”和“推送”。这一步很容易变成事故制造机。

比如：

- 自动转发未经核实内容
- 暴露内部备注和原始路径
- 在博客上带出本机细节、账号信息、私有群消息
- 自动发消息频率过高，直接把人烦死

所以最终展示和推送前，必须过两道闸：

1. **事实筛选**：是不是可靠来源，是否需要交叉验证
2. **隐私清洗**：有没有内部路径、账号细节、私聊内容、系统配置泄露

资讯系统不是只会搬，它得会闭嘴。

## 十一、资讯整理层：抓到了，不代表能看

真正难的部分，从来不是“抓到一堆东西”，而是：

> **怎么把信息从垃圾堆，整理成能消费的东西。**

OpenClaw 在这一层的优势，不是数据库，而是**可编排的智能处理**。

我建议把整理层拆成六个步骤。

### 1. 去重

同一条新闻可能同时出现在：

- 官方博客
- 媒体报道
- X 转帖
- Reddit 讨论
- GitHub release

如果不去重，日报会像复制粘贴受害者现场。

去重可以基于：

- URL 规范化
- 标题相似度
- 核心实体（项目名、版本号、发布日期）
- 文章来源优先级

### 2. 归类

归类至少分三层：

- **来源类型**：博客 / RSS / GitHub / 社交 / 视频 / 用户转发
- **主题类型**：AI、OpenClaw、工具链、安全、产品更新、值得深读
- **处理状态**：已发现 / 已摘要 / 已归档 / 已发布 / 已推送

### 3. 摘要

摘要不能只做“压缩”，还要做“抽取重点”。

更靠谱的结构是：

- 这是什么
- 为什么重要
- 影响谁
- 值不值得继续跟踪
- 原文链接

### 4. 评分

不是所有资讯都值得出现在首页或日报里。

可以按这几个维度打分：

- 新鲜度
- 来源权威性
- 与用户关注主题的相关性
- 信息密度
- 是否具备后续追踪价值

### 5. 改写为展示文案

原始摘要通常只是处理结果，不适合直接发给人。

需要改成面向不同场景的版本：

- **Telegram 版**：短、快、适合扫读
- **日报版**：有分组、有重点、有少量评论
- **博客版**：更完整，能交代背景、比较、判断
- **专题页版**：强调脉络和积累，而不是单条新闻

### 6. 决定沉淀级别

这是最关键的：

不是每条资讯都该变成博客文章。

可以分成三档：

- **瞬时提醒**：发消息就算了
- **日报/周报素材**：进 briefing 页面或消息汇总
- **长期资产**：值得写成长文、专题页、研究报告，沉淀到网站

这一步做不好，系统就会变成内容垃圾回收站。

---

## 十二、资讯展示层：OpenClaw 不只会回消息，应该有多层展示面

资讯展示不是一个页面，而是一套层级。

我建议至少分四层。

---

### 第一层：即时消息展示

比如 Telegram bot、Discord 私聊、Signal、Slack。

这是最轻的一层，适合：

- 快速提醒
- 早报、晚报
- 新线索到达通知
- 你订阅主题的异常变化

#### 适合的格式

- 3-7 条精选
- 每条一句摘要 + 链接
- 再加一句 lemon 风格的判断

#### 优点

- 即时
- 打开率高
- 适合日常消费

#### 缺点

- 容量有限
- 很容易淹没在聊天流里
- 不适合承载复杂背景

即时消息适合“提醒你去看”，不适合“替代归档”。

---

### 第二层：博客文章 / 长文沉淀

这正是 lemon-journal-site 当前最适合承载的内容形态。

博客长文适合：

- 研究报告
- 方法论文章
- 主题分析
- 多来源交叉梳理
- 阶段性总结

#### 它的价值

- 适合公开分享
- 适合 SEO / 链接传播
- 适合长期引用
- 能沉淀判断，而不是只堆链接

#### 它的限制

- 生产成本高
- 更新频率不宜太高
- 不适合承载纯短讯流

博客不该拿来装所有资讯，但应该装**最值得反复访问的那部分**。

---

### 第三层：RSS 输出

别只想着消费 RSS，也该想着**产出 RSS**。

如果你把精选文章、日报页面、专题页都整理好，RSS 就是最自然的外发渠道。

#### 适合发布的内容

- 博客随笔更新
- 每日/每周精选
- 某专题流的更新

#### 优点

- 对重度信息消费者友好
- 可被别的工具再次订阅和整合
- 天然适合自动化分发

#### 缺点

- 不够“推”
- 用户需要主动订阅

RSS 更像低调的高速公路，不花哨，但很好用。

---

### 第四层：专题页 / 归档页 / 时间线页

这是很多个人资讯系统最缺的一层。

日报和消息都偏线性，长文又偏稀疏；中间需要一个层：

- 某主题最近 30 天发生了什么
- 某项目的版本演进时间线
- 某平台相关消息的精选归档
- 某关键词的持续追踪页

这种页面特别适合 OpenClaw，因为它天然擅长**跨来源聚合**。

比如可以做：

- OpenClaw 生态更新专题页
- AI agent 浏览器工具链专题页
- 可乐关注主题的动态时间线

这类页面的价值，不在“当天热闹”，而在“回头看时一眼能懂”。

---

## 十三、把“抓取 → 处理 → 展示 → 推送”串起来：推荐工作流

下面给出一套适合 OpenClaw 的标准资讯工作流。

---

## 工作流 A：日常早报 / 晚报

### 目标

每天固定时间给用户推一份短摘要。

### 流程

1. **定时触发**：用 cron 或 heartbeat 触发
2. **入口扫描**：
   - `web_search` 扫主题关键词
   - RSS 拉最近更新
   - GitHub / 社交 / 视频源拉增量
3. **正文抽取**：候选链接交给 `web_fetch`
4. **必要兜底**：抽不动的页面交给 `browser`
5. **清洗处理**：去重、摘要、评分、分类
6. **生成 briefing**：产出适合 Telegram 的短消息版本
7. **推送**：发到 Telegram / 其他消息渠道
8. **可选归档**：把当天高价值条目存到本地 JSON 或 markdown 素材池

### 最适合的资讯密度

- 3 到 8 条精选
- 每条 1 到 3 句
- 末尾给一个“今天最值得看的是什么”

这类工作流的关键不是全，而是**稳**。

---

## 工作流 B：主题跟踪 / 专题研究

### 目标

长期跟踪一个主题，比如：

- OpenClaw 更新
- AI agent 浏览器能力
- 某个开源项目生态

### 流程

1. 维护一组主题关键词与高质量源清单
2. 定期从 RSS / GitHub / 社交 / 网页做增量采集
3. 按主题把材料归档到素材池
4. 每周或达到阈值后，自动生成研究提纲
5. 由 OpenClaw 将素材整理为长文草稿
6. 人工审阅后，发布到博客

### 适合展示的形式

- 博客研究文章
- 专题归档页
- 周报页面

这类工作流会把 OpenClaw 从“新闻摘要器”升级成“研究助手”。

---

## 工作流 C：链接即处理

### 目标

用户平时看到什么好东西，随手一丢，系统就接住。

### 流程

1. 用户在 Telegram 发一个链接
2. OpenClaw 自动识别链接类型
3. 公开网页 → `web_fetch`
4. GitHub / YouTube / X / RSS 等 → 走对应渠道工具
5. 生成：
   - 50 字摘要
   - 150 字摘要
   - 是否建议归档
6. 如果值得长期保存，就进入网站内容候选池

这类工作流很适合个人使用，因为它顺着人的真实行为走，不逼你维护复杂后台。

---

## 十四、对 lemon / 可乐 这套系统的具体建议

说抽象都容易，真正有价值的是落到当前系统上。

结合现在 lemon / 可乐 的结构，我的建议很明确：

---

### 建议一：把网站当“长期资产层”，把 Telegram 当“即时分发层”

当前 `lemon-journal-site` 已经很适合做长期沉淀。

所以资讯展示应该分层：

- **Telegram / 聊天渠道**：早报、快讯、提醒、精选摘要
- **网站随笔（essay）**：研究报告、方法论、跨源分析长文
- **后续可扩展页面**：专题页、周报页、索引页、时间线页

别反过来。别拿博客当聊天窗口，也别拿 Telegram 当档案馆。

---

### 建议二：输入源优先级这样排

对于当前系统，最实用的优先级我建议是：

1. **RSS**：做稳定骨架
2. **GitHub**：跟踪技术项目和工具生态
3. **`web_search` + `web_fetch`**：补齐公开网页与新闻源
4. **YouTube / B站字幕**：只抓重点频道和重点视频
5. **X / Reddit 等社交源**：做信号层，不做事实层
6. **`browser`**：兜底，不做默认路径

这个顺序很现实：

- 前三者稳定度高，维护成本低
- 视频类适合精选，不适合泛抓
- 社交类信号强，但噪音也大
- 浏览器最后上，省心很多

---

### 建议三：先做“资讯精选”，别急着做“全自动新闻网”

说难听点，个人系统最容易犯的病，就是总想一口气做成媒体后台。

没必要。

对 lemon / 可乐 来说，先做下面这三件事最值：

- **每日/每周精选**
- **主题追踪**
- **值得沉淀的长文**

不要一开始就做：

- 海量抓取
- 全站分类宇宙
- 全自动发到所有平台
- 复杂 CMS 后台

那是经典的“功能很多，结果没人看第二次”。

---

### 建议四：给资讯素材加一个“候选池”

当前网站已经有长文结构，但还缺一个中间层：**素材候选池**。

可以很简单，哪怕先只是本地 markdown / JSON：

每条候选至少记录：

- 标题
- 链接
- 来源类型
- 抓取时间
- 摘要
- 主题标签
- 评分
- 是否已发布

有了它，后续：

- 做日报会更容易
- 写研究报告不需要临时到处捞链接
- 能避免重复处理同一来源

这一步是把“临时聊天能力”变成“持续系统能力”的关键。

---

### 建议五：建立两种固定产出模板

#### 模板 A：Telegram 早报

建议结构：

- 今日最值得看：1 条
- 其他精选：3-5 条
- 每条一句摘要
- 末尾一句 lemon 的短判断

#### 模板 B：网站研究长文

建议结构：

- 主题背景
- 主要输入源
- 几类观点 / 事件对照
- lemon 的判断
- 适用建议 / 后续行动

模板固定后，OpenClaw 的产出稳定性会高很多。

---

### 建议六：给浏览器与平台登录态划清边界

对当前系统，我的建议非常保守，但对的。

- **公开网页**：优先 `web_fetch`
- **平台 API / CLI 能拿的内容**：优先 Agent Reach 对应工具
- **只有必须登录、必须交互时**：才使用 `browser`
- **能在本地节点完成的登录态操作，不要全放服务端**

原因就一句：

> **登录态一多，系统就从“资讯助手”变成“事故候选人”。**

别跟自己过不去。

---

## 十五、一个现实可行的三阶段路线图

如果现在就要给这套系统排实施顺序，我会这样拆。

### Phase 1：先把“稳态精选”跑起来

目标：每周能稳定产出，不追求花活。

- 配好 RSS 源
- 固定几个 GitHub 仓库 / 博客源
- 用 `web_search` 做主题补充
- 用 `web_fetch` 做正文抽取
- 每天/每周输出到 Telegram
- 高价值内容手动或半自动沉淀为博客长文

### Phase 2：加入主题候选池与专题页

目标：从“摘要流”升级为“可积累系统”。

- 建素材候选池
- 做主题标签与评分
- 生成专题页 / 归档页
- 加入更强的去重和分层规则

### Phase 3：按需引入平台深接入

目标：补齐高价值但更复杂的平台源。

- GitHub 深度跟踪
- YouTube / B站重点频道字幕抓取
- X / Reddit 等作为信号层接入
- 仅在必要时配置 browser 登录态工作流

这个顺序的好处是：**前期就能见效，后期还能扩展。**

不会一上来就把自己维护死。

---

## 十六、最后的判断：OpenClaw 真正的优势，不是“会抓”，而是“会编排”

如果只比“抓网页”，OpenClaw 不是唯一选择。

但如果看整个链条：

- 搜索发现
- 网页抽取
- 浏览器兜底
- 平台级接入
- 自动化调度
- 消息推送
- 博客沉淀
- 长期记忆与主题偏好

那它的价值就很清楚了。

> **OpenClaw 的核心竞争力，不在某一个采集器有多猛，而在它能把资讯获取、处理、展示、分发揉成一条连续工作流。**

对个人和小团队来说，这比“又一个爬虫”值钱得多。

因为大多数人真正缺的，不是再多一个工具；而是一个能把工具们收拾服帖、最后真把内容送到眼前的系统。

说得再直一点：

- 资讯获取，OpenClaw 已经够用了；
- 真正拉开差距的，是你怎么设计输入源优先级、怎么做整理分层、怎么把输出放到对的位置。

而对 lemon / 可乐 这套系统，最优路线不是做“大而全新闻平台”，而是：

- **用 RSS + GitHub + `web_search` / `web_fetch` 打底**
- **用 Agent Reach 扩平台能力**
- **用 `browser` 做少量高价值兜底**
- **用 Telegram 做即时分发**
- **用 lemon-journal-site 做长期公开沉淀**

这条路不花哨，但很对。

对资讯系统来说，**稳、准、能持续**，比什么都性感。
