很多人把 OpenClaw 当成“会聊天、会调工具的助手”，把 qmd 当成“能渲染成 HTML / PDF 的 Markdown 文件”。

这两个理解都没错，但都太保守了。

真把它们放在一起看，会发现它们组合出来的，不是一个“写文档小技巧”，而是一条很像样的**研究生产线**：

- OpenClaw 负责找资料、抓网页、整理结构、补充上下文、调度自动化；
- qmd 负责把结果变成**可复现、可渲染、可长期维护**的正式报告；
- 最后还能继续输出到 HTML、PDF、演示文稿，甚至变成网站内容资产。

说白了：

> **OpenClaw 解决“怎么把材料搞到手、整理成形”，qmd 解决“怎么把结果稳定地表达、渲染、复用、留档”。**

这篇研究报告就研究这一件事：**qmd 到底该怎么和 OpenClaw 配合，才能从“临时聊天产物”升级成“长期可用的研究工作流”。**

---

## 一、先下结论：这套组合值不值得上

值，而且挺值。

但不是因为它“先进”，而是因为它把两种本来分裂的能力接上了：

- 聊天式获取与探索
- 文档式沉淀与复现

OpenClaw 强在动态、交互、编排；qmd 强在结构、渲染、计算、归档。

这意味着你可以把原本散落在聊天记录里的东西，慢慢收敛成：

- 一篇正式研究报告
- 一份可重复运行的分析文档
- 一套带参数的模板
- 一个可以持续更新的专题项目

一句话版本：

> **如果你只是想问一个问题然后看一眼答案，qmd 没必要。**
>
> **如果你要的是“以后还能改、还能复用、还能渲染、还能给别人看”的成果物，那 qmd 非常合适。**

## 二、什么是 qmd，它为什么适合接 OpenClaw

qmd 就是 Quarto Markdown 文件，底层是 Pandoc 风格 Markdown，再叠加 YAML front matter、代码块执行、交叉引用、图表、引用、表格、公式、div 容器这些能力。

它适合和 OpenClaw 配合，核心不是“文件后缀不同”，而是下面这几件事：

### 1. 它天然支持“正文 + 代码 + 输出”写在一起

这很关键。

因为很多研究报告不是纯文本，它往往还会带上：

- 数据处理代码
- 图表生成代码
- 运行参数
- 自动生成的摘要或中间结果

OpenClaw 可以负责把资料、结构、解释写进去；qmd 则负责把这些内容和计算结果放在一个文件里，不用东一块西一块地拼。

### 2. 它适合多格式输出

同一份 qmd，通常可以渲染成：

- HTML
- PDF
- Word
- reveal.js 幻灯片
- 网站页面
- 书籍 / 项目文档的一部分

这意味着 OpenClaw 整理出的内容，不会被锁死在聊天窗口里。

### 3. 它支持参数化与项目化

当你不是只写“一篇”，而是想写“很多篇同结构的报告”时，qmd 就开始发力。

比如：

- 每周行业追踪报告
- 某主题的固定研究模板
- 某项目的版本跟踪分析
- 定期自动汇总的周报 / 月报

OpenClaw 可以负责拉材料，qmd 可以负责套模板和稳定输出。

### 4. 它比“纯 Markdown”更像正式生产格式

普通 Markdown 适合记笔记。

qmd 更适合做**带工程约束的正式文档**，因为它可以比较认真地处理：

- 目录
- 编号
- 交叉引用
- 图表标题
- 代码执行选项
- 输出格式差异
- 中间文件保留

这对研究报告很重要。你总不能每次都靠“复制粘贴玄学”和“希望浏览器别炸”。

## 三、OpenClaw 在这条链路里最适合干什么

别把 OpenClaw 想成“自动写作文机”。那是对它最大的浪费。

它更适合做的是前中段工作，也就是把报告真正难做的部分扛起来。

### 1. 选题与问题拆解

研究报告最怕一上来就开写。那基本等于裸奔。

OpenClaw 很适合先做：

- 主题范围收窄
- 核心问题拆解
- 读者对象判断
- 输出形式选择
- 报告结构草拟

也就是先把“写什么”从一坨雾里拽出来。

### 2. 信息获取

这一层正好是 OpenClaw 的主场。

它可以按成本和复杂度分层拿信息：

- `web_search`：找线索、找候选来源
- `web_fetch`：抓公开网页正文
- `browser`：处理动态页面、复杂页面、必要交互
- Agent Reach 工具链：补 GitHub、视频、社交、RSS 等平台型来源

所以它特别适合做研究的“资料入口层”。

### 3. 初步整理

拿到资料不等于报告成了。

OpenClaw 很适合先做一轮结构化整理，比如：

- 去重
- 摘要
- 主题归类
- 观点分组
- 时间线梳理
- 风险点标记
- 术语统一

这一步做完，qmd 才不至于变成漂亮外壳里塞一堆垃圾。

### 4. 草稿生成

到这里，OpenClaw 才应该开始“写”。

但注意，最合理的写法不是一次性整篇喷出来，而是分层产出：

- 标题与摘要
- 章节结构
- 每章关键结论
- 例子 / 引用 / 图表建议
- 附录或方法说明

这样产出的内容，比较适合直接填进 qmd 模板里。

### 5. 自动化调度

如果你的报告不是一次性的，而是周期性的，OpenClaw 的 cron、消息、记忆、站点工作流编排就很有价值。

比如：

- 每周抓一次主题增量
- 每天整理素材候选池
- 每月生成一版报告草稿
- 完成后发到 Telegram 提醒你审稿

这时候 OpenClaw 不只是“帮你写文档”，而是“帮你维持一条文档生产线”。

## 四、qmd 在这条链路里最适合干什么

OpenClaw 擅长动，qmd 擅长稳。

它最适合承担四类职责。

### 1. 作为正式成果容器

聊天记录再聪明，也不该当最终交付。

qmd 更适合做最终产物的承载层，因为它：

- 有明确文件边界
- 能版本管理
- 能 diff
- 能审阅
- 能渲染成正式格式

这比“在聊天里翻第 27 条消息找结论”强太多了。

### 2. 作为可复现计算文档

如果报告里带代码执行、图表、表格统计、参数化分析，那 qmd 的优势会很明显。

Quarto 支持在文档里控制代码执行选项，比如：

- `echo`
- `eval`
- `warning`
- `error`
- `include`
- `output: asis`
- `keep-md`
- `keep-ipynb`

这意味着你可以决定：

- 哪些代码展示给读者看
- 哪些只运行不展示
- 哪些输出原样插入正文
- 是否保留中间产物便于调试

这就不是“会写 Markdown”了，这是在做有规矩的可复现文档。

### 3. 作为模板系统

真正实用的研究流程，不是“终于写完这一篇”，而是“下一篇能不能更快”。

qmd 非常适合被设计成模板。

比如固定一个研究模板：

- 背景
- 问题定义
- 方法
- 资料来源
- 核心发现
- 风险与限制
- 建议
- 附录

OpenClaw 每次只要把内容塞进去、补齐细节、调整逻辑，就能稳定产出。

### 4. 作为跨格式发布中间层

你可以把 qmd 当成“中间标准件”。

也就是说：

- OpenClaw 负责把内容生成到 qmd
- Quarto 负责把 qmd 渲染成 HTML / PDF / Word / Slides
- 网站、邮件、消息渠道再拿这些结果继续分发

这样链路就清楚了，不会所有东西都糊在聊天窗口里发烫。

## 五、最推荐的使用姿势：把 OpenClaw 当研究助手，把 qmd 当报告底稿

这是我最推荐的姿势，而且很适合当前 lemon / 可乐 的系统。

### 不推荐的姿势

- 让 OpenClaw 直接把最终版全文一次吐完
- 完全不设模板，想到哪写到哪
- 把聊天记录直接当报告成品
- 把 qmd 只当“另一个 Markdown 文件”来用

这些做法不至于完全不能用，但很容易出现三种翻车：

1. 结构漂移
2. 事实引用混乱
3. 后续无法维护

### 推荐的姿势

更稳的流程应该是：

1. **先定报告目标**：研究什么、给谁看、输出什么格式
2. **OpenClaw 拉材料**：搜索、抓取、整理、提纲
3. **用 qmd 模板承接**：把结构固定下来
4. **OpenClaw 分章节填充**：而不是一次整篇乱喷
5. **需要计算就放代码块**：把图表和统计真正纳入文档
6. **最后再渲染和发布**：HTML、PDF、网站各走各的

一句话：

> **聊天负责探索，qmd 负责定稿。**

这才是两边都不受委屈的分工。

## 六、一个现实可用的 qmd 报告骨架

下面给一份很适合和 OpenClaw 配合的 qmd 骨架。

```yaml
---
title: "OpenClaw 研究报告标题"
author: "lemon"
date: 2026-03-14
format:
  html:
    toc: true
    toc-depth: 3
    number-sections: true
    code-fold: false
    embed-resources: true
  pdf:
    toc: true
    number-sections: true
execute:
  echo: false
  warning: false
  error: false
  keep-md: true
---
```

正文结构建议这样：

```markdown
# 摘要

一句话结论 + 三到五条核心发现。

# 研究问题

这篇报告到底在回答什么问题。

# 背景与上下文

为什么现在值得研究这件事。

# 方法与资料来源

用了哪些来源、怎么筛选、有什么限制。

# 核心发现

## 发现一
## 发现二
## 发现三

# 风险、限制与反例

哪些地方不能过度下结论。

# 结论与建议

给决策用的结论，不要又绕回空话。

# 附录

链接、补充样例、原始表格、代码说明。
```

这个骨架的好处是：

- 结构清楚
- 适合 OpenClaw 分块填充
- 后续能接图表、表格、引用
- HTML / PDF 都比较好看

不会写成一坨没骨头的长聊天记录。

## 七、如果要让 OpenClaw 真参与 qmd 生产，最适合的分工是什么

下面这套分工最靠谱。

### 阶段 A：资料整理前

OpenClaw 负责：

- 明确主题边界
- 给出研究问题列表
- 列出资料获取计划
- 产出初版章节提纲

qmd 先不急着写正文，只先起一个骨架文件。

### 阶段 B：资料获取与归档

OpenClaw 负责：

- 搜索来源
- 抽取正文
- 汇总摘要
- 标出冲突信息
- 给出来源可信度判断

qmd 负责承接：

- 方法部分
- 资料来源列表
- 引用占位
- 附录素材区

### 阶段 C：章节填充

OpenClaw 负责分章节生成内容：

- 摘要
- 背景
- 核心发现
- 风险与建议

qmd 负责把这些内容固定成结构化文档，而不是临时聊天结果。

### 阶段 D：计算与图表

如果需要：

- Python / R / Julia 代码块放进 qmd
- 图表由 Quarto 渲染
- 文本说明由 OpenClaw 辅助润色

这一步尤其适合“研究 + 数据”的场景。

### 阶段 E：最终发布

- 站内阅读 → HTML
- 对外发送或归档 → PDF / Word
- 后续引用 → 保留 qmd 源文件

这才叫完整闭环。

## 八、一个更像样的 OpenClaw + qmd 工作流示例

下面用一个具体例子来说明。

假设你要写一篇：

> “OpenClaw 如何用于持续资讯研究与自动分发”

更合理的流程不是“让模型直接写 5000 字”，而是：

### Step 1：OpenClaw 先做研究拆解

先得到：

- 研究目标
- 目标读者
- 需要覆盖的能力边界
- 可能的反例和风险点

### Step 2：OpenClaw 把资料抓回来

例如：

- 文档页
- 官方仓库
- 博客文章
- 已有内部实践记录
- 配套工具说明

### Step 3：OpenClaw 输出“结构化草稿包”

不是直接整篇文章，而是：

- 标题建议
- 摘要候选
- 章节提纲
- 每章要点
- 值得保留的例子
- 风险提示

### Step 4：把这些内容塞进 qmd 模板

qmd 开始像这样：

```yaml
---
title: "OpenClaw 如何用于持续资讯研究与自动分发"
format:
  html:
    toc: true
    number-sections: true
execute:
  echo: false
  warning: false
---
```

然后正文各章逐步填进去。

### Step 5：如果有统计或图表，再接代码块

比如统计不同资讯来源的稳定性：

```python
#| echo: true
#| fig-cap: "不同资讯来源的维护成本与稳定性"

import matplotlib.pyplot as plt

sources = ["RSS", "GitHub", "web_fetch", "browser"]
stability = [9, 8, 7, 4]
maintenance = [2, 3, 4, 8]

plt.figure(figsize=(8, 4))
plt.plot(sources, stability, label="稳定性")
plt.plot(sources, maintenance, label="维护成本")
plt.legend()
plt.tight_layout()
plt.show()
```

这样最后出来的不是“模型说了点道理”，而是一份真的能交付的研究报告。

## 九、qmd 最适合承接哪些 OpenClaw 产物

不是所有 OpenClaw 结果都要进 qmd。

真这么干，文档会很快变成垃圾仓库。

我建议只让下面这些内容进入 qmd：

### 1. 稳定结论

比如：

- 对某工具的能力判断
- 对某方案的工作流建议
- 对某系统边界的研究结论

这些适合长期保留。

### 2. 可复现方法

比如：

- 如何抓数据
- 如何跑分析
- 如何生成图表
- 如何渲染输出

这些是下次还能用的东西。

### 3. 正式引用和例子

像：

- 官方文档关键信息
- 示例代码
- 表格、图示、引用段落

这些适合沉淀进正式文档。

### 4. 附录性质的材料

比如：

- 原始链接列表
- 参数说明
- 术语表
- 样例输出

这些也很适合进 qmd。

### 不建议直接进 qmd 的

- 聊天里的来回试探
- 很水的 brainstorming 废话
- 没核验的临时判断
- 对用户来说没价值的中间啰嗦过程

报告不是聊天转储站，真不是。

## 十、这套组合最适合的场景

### 1. 周期性研究报告

最典型。

比如：

- 每周 AI 工具链观察
- 每月 OpenClaw 生态更新
- 某主题长期跟踪报告

OpenClaw 负责增量搜集，qmd 负责固定模板输出。

### 2. 技术方案评估

比如：

- 某工具值不值得接入
- 某架构路线怎么取舍
- 某自动化方案的边界和风险

这种报告需要结构、例子、对比、结论，非常适合 qmd。

### 3. 数据分析类文档

如果要带代码、图表、表格，这就是 Quarto 主场。

OpenClaw 可以帮你：

- 解释思路
- 写初版代码
- 整理结论
- 解释图表

qmd 则负责把运行和输出固定下来。

### 4. 面向网站或公开归档的长文

这点跟当前系统尤其契合。

你现在已经有站点作为长期资产层，那么研究报告最适合先在 qmd / 结构化 Markdown 中完成，再决定怎么进站展示。

## 十一、这套组合不适合什么场景

### 1. 临时闲聊式问题

问一句、看一眼、结束。

这种就别上 qmd 了，属于拿火箭发豆腐。

### 2. 完全没有稳定结构的开放式发散

如果问题还处于“我甚至不知道我要问啥”的阶段，先聊天探索更合适。

等结构成形了，再落 qmd。

### 3. 需要极重交互但不需要留档的任务

比如纯浏览器点点点、一次性后台操作、临时网页验证。

这些让 OpenClaw 干就行，没必要硬塞进 qmd。

### 4. 指望一键全自动出终稿

能不能做一点自动化？能。

能不能完全无人监管、资料自动抓、结论自动写、格式自动发、永不翻车？

醒醒，别做梦了。

研究报告这种东西，尤其只要涉及判断、取舍、引用和对外展示，就应该保留人工审阅。

## 十二、实际落地时最容易翻车的地方

### 1. 把 OpenClaw 的草稿当最终事实

模型擅长整理，不等于它自动拥有事实权威。

所以：

- 外部资料要核验
- 关键结论要回看来源
- 引用要明确边界

别让文档写得很像那么回事，里面却掺了幻觉和脑补。

### 2. 把 qmd 模板设计得太花

别一开始就堆：

- 十几种输出格式
- 一堆自定义 CSS
- 一堆复杂 filters
- 一堆宏与 includes

这很容易从“研究流程”走向“配置地狱”。

先能稳定出报告，再谈花活。

### 3. 结构没定就开始补正文

这会导致：

- 前后重复
- 结论漂移
- 论证层次乱掉

先让 OpenClaw 把提纲和问题边界整理好，再填内容，效率高太多。

### 4. 代码执行和文本叙述脱节

qmd 很适合代码 + 叙述混排，但前提是两者真的有关。

别搞成：

- 文本在说 A
- 图表在证明 B
- 代码跑出来的是 C

读者看了只会想骂人。

### 5. 把所有中间过程都写进正式文档

正式报告不需要展示全部试错痕迹。

OpenClaw 的探索过程，可以留在聊天、素材池、笔记区；
qmd 只保留对读者真正有用的那层。

---

## 十三、对 lemon / 可乐 这套系统的具体建议

这里不讲空的，直接讲落地。

### 建议一：研究报告继续优先沉淀到站点项目里

这条不用拐弯。

当前系统已经有 `lemon-journal-site` 作为长期内容资产层，所以凡是像样的研究报告，都应该优先沉淀到这个项目，而不是只存在本地零散文件里。

这能带来三件好事：

- 有长期 URL
- 可持续维护
- 不是一次性聊天产物

### 建议二：短期先用“结构化 Markdown 承载 + qmd 作为模板源”

目前站点内容集合主要吃 `.md`，不是直接吃 `.qmd`。

所以最现实的做法不是硬改整站内容管线，而是：

- **研究流程内部**：可以用 qmd 模板来组织正式报告
- **站点发布层**：再转成站点能稳定接住的 Markdown / 页面内容

这比一上来就改站点构建链路稳得多。

### 建议三：建立一个 qmd 模板目录

建议后续加一个类似这样的目录：

```text
projects/lemon-journal-site/reports/templates/
```

里面放：

- 研究报告模板
- 周报模板
- 技术评估模板
- 数据分析模板

这样 OpenClaw 每次生成内容时，就不是凭空乱写，而是有骨架可套。

### 建议四：OpenClaw 产出应以“章节包”形式喂给 qmd

我很建议以后别再让模型一次吐整篇终稿。

更稳的是：

- 标题候选
- 摘要候选
- 目录
- 每章结论
- 示例代码
- 图表说明
- 附录链接

然后再由 qmd 模板承接。

这会让报告质量明显更稳。

### 建议五：需要自动化的地方，用 OpenClaw；需要可复现的地方，用 qmd

这个分工最清楚：

- 自动搜索、抓取、归档、提醒 → OpenClaw
- 正式结构、代码执行、图表渲染、导出格式 → qmd

别反着来。反着来就会越来越拧巴。

## 十四、一个最小可行方案（MVP）

如果现在就要开始，而不是继续脑内装修，我建议分三步。

### Phase 1：先固定模板

做一份通用研究报告 qmd 模板，字段尽量少但够用：

- 标题
- 摘要
- 研究问题
- 方法
- 核心发现
- 风险与限制
- 结论与建议
- 附录

### Phase 2：让 OpenClaw 先输出“可贴进去的内容块”

每次研究任务，先让 OpenClaw 输出：

- 提纲
- 每章要点
- 关键例子
- 参考链接列表

而不是先追求整篇终稿。

### Phase 3：再考虑自动渲染或站点接入

等模板稳了、内容质量稳了，再去做：

- 自动导出 HTML / PDF
- 同步到站点
- 周期性报告生成
- 更复杂的图表和参数化

这个顺序很土，但很对。

## 十五、最后的判断：qmd + OpenClaw 的真正意义，不是“更会写”，而是“更能留下来”

大多数聊天式产出最大的问题，不是当下不好看，而是**留不下来**。

- 过几天找不到
- 过几周不敢改
- 过几个月无法复用
- 过一阵子连自己都不知道当时怎么得出那个结论

而 qmd 配合 OpenClaw，解决的恰好就是这个问题。

它把原本一次性的研究劳动，慢慢变成：

- 有结构的
- 可复现的
- 可渲染的
- 可维护的
- 能沉淀成长期资产的

这才是它真正性感的地方。

说得再直白点：

> **OpenClaw 让研究更省力，qmd 让研究不白干。**

如果只是问一句答案，聊天窗口够了；
如果要做一份以后还会回来看的报告，那就别偷懒，老老实实上 qmd。

这不是仪式感，这是给未来的自己少挖坑。
