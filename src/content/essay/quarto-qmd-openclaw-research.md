先说清楚：这里的 **qmd**，不是 Quarto 的 `.qmd` 文档格式，
而是 GitHub 上的 **[`tobi/qmd`](https://github.com/tobi/qmd)** —— 一个偏 agent 工作流的**本地文档搜索引擎**。

这玩意儿干的事很直接：

- 给你的 notes、docs、meeting transcripts、knowledge base 建索引；
- 支持关键词搜索、语义搜索、混合搜索；
- 还能做 LLM rerank；
- 全程尽量本地跑；
- 顺手暴露成 CLI、SDK 和 MCP server，方便被别的 agent 系统接上。

而 OpenClaw 的强项，是调度工具、管理多通道输入输出、跑自动化、承接对话、串起工作流。

所以这两个东西放在一起，真正有意思的点不是“能不能接”，而是：

> **qmd 负责把本地文档变成可检索知识层，OpenClaw 负责把这个知识层真正接进代理行为。**

一个管“找得到”，一个管“用得起来”。

这篇报告就研究这件事：**tobi/qmd 到底适不适合和 OpenClaw 一起用，应该接在哪一层，值不值得上。**

---

## 一、先下结论：值，而且挺对路

如果你的问题是：

- 我有一堆 Markdown、文档、笔记、会议记录、项目资料；
- 我不想全丢给云端；
- 我希望 agent 能搜得到、拿得准、读得全；
- 我希望这个检索层能被 CLI / MCP / 代码统一调用；

那 qmd 跟 OpenClaw 的组合是**非常顺手**的。

但也别神化。

它适合的是：

- **中小规模本地知识库检索**
- **agent 辅助阅读 / 回答 / summarization 工作流**
- **以 Markdown / 文本文档为主的知识资产**
- **强调本地运行和隐私控制的场景**

它不适合的是：

- 超大规模企业搜索基础设施
- 权限体系极复杂的知识门户
- 海量在线实时数据接入
- 指望“一装即神谕”的全自动真相机

一句话：

> **qmd 很适合做 OpenClaw 的“本地知识检索底座”，但不该被幻想成万能记忆大脑。**

## 二、qmd 到底是什么

按仓库描述，qmd 是一个 **on-device search engine for everything you need to remember**。

翻成人话，就是：

- 它帮你把本地文档建成索引；
- 然后你可以用关键词、自然语言、混合检索去找；
- 最后把结果交给 agent 或人继续消费。

它的能力大致分三层：

### 1. 关键词检索

也就是 BM25 这一类传统全文搜索。

优点：
- 快
- 稳
- 可解释
- 对精确术语、函数名、报错信息、文件名特别有用

缺点：
- 不懂语义绕弯
- 关键词写偏了容易漏

### 2. 向量语义检索

也就是 embed 后按语义相近找内容。

优点：
- 不依赖精确词面
- 适合“我大概知道意思，但记不清原话”这种场景
- 对笔记、会议记录、概念类材料很有帮助

缺点：
- 要先生成 embeddings
- 结果解释性不如关键词检索直观
- 如果文档切分和上下文做得烂，语义命中会发飘

### 3. 混合检索 + rerank

这是它更像“agent 工具”的地方。

仓库说明里，qmd 支持：

- lex / vec / hyde 等 typed sub-queries
- RRF 合并
- LLM reranking
- context metadata

也就是说，它不是简单做一个向量数据库壳子，而是认真在搞一条**面向 agent 的混合检索链路**。

这就很关键了。

因为 OpenClaw 真要接一个检索工具，不是只要“能搜”，而是要：

- 给得出还不错的候选
- 结果尽量别太散
- 能把上下文一起交出来
- 最好还能继续 retrieve 原文

qmd 在这些点上，方向是对的。

## 三、qmd 为什么适合接 OpenClaw

OpenClaw 本身已经有自己的记忆机制，也有文件读写、网页抓取、消息分发、定时任务、浏览器、节点、子代理等能力。

那为什么还需要 qmd？

因为它补的是另一层：

> **不是“长期记住什么”，而是“在大堆本地文档里，快速找出和当前问题最相关的材料”。**

这两者不是一回事。

### OpenClaw 的 memory 更像什么

更像：

- 个人偏好
- 长期稳定事实
- 项目级经验总结
- 少量高价值、人工筛选过的记忆

也就是“值得长期保留的 distilled memory”。

### qmd 更像什么

更像：

- 全量笔记库
- 会议纪要堆
- 文档目录树
- 项目说明集合
- 原始知识资产的检索层

也就是“可搜索的文档宇宙”。

这就是为什么它们不冲突，反而互补。

- **memory** 解决“该记住什么”
- **qmd** 解决“需要时去哪找”
- **OpenClaw** 解决“什么时候找、找完怎么用、结果怎么送出去”

这分工非常健康。

## 四、qmd 最适合接在哪一层

我不建议一上来就把 qmd 神圣化成系统中心。

更合理的接法，是把它放在下面几个层位之一。

### 1. 当本地知识检索 CLI

这是最低摩擦、最现实的接法。

OpenClaw 需要搜本地资料时，直接调用：

- `qmd search`
- `qmd vsearch`
- `qmd query`
- `qmd get`
- `qmd multi-get`

这种接法优点很明显：

- 不用先改 OpenClaw 配置体系
- 先能试工作流
- 很适合验证索引质量和资料组织方式
- 失败成本低

缺点也明显：

- 还是命令式集成
- 结果格式和后续消费要自己兜
- 没有 MCP 那么优雅

但说实话，**MVP 阶段就该这么干**。别一上来就造教堂。

### 2. 当 MCP 服务

qmd 直接暴露 MCP server，这就很适合 agent 系统接入。

它暴露的工具包括：

- `query`
- `get`
- `multi_get`
- `status`

如果 OpenClaw 侧后续通过 MCP/桥接方式消费这套能力，那会更自然：

- 结构化调用
- 更少解析胶水代码
- 更方便复用
- 更适合让多个 agent 共享同一检索服务

尤其 qmd 还支持 **HTTP transport**，能做长驻服务，避免每次重载模型。这点对 agent 工作流非常友好。

### 3. 当研究 / 报告的素材索引层

这个特别适合你现在这套系统。

因为你已经有：

- 站点项目作为长期内容资产层
- OpenClaw 负责日常交互和生产
- 本地 workspace 里有各种 notes / memory / docs / reports

这时候 qmd 非常适合做：

- 素材候选库索引
- 历史研究报告检索
- 项目文档检索
- 日志 / meeting notes 检索

这样 OpenClaw 在写新报告时，不用傻乎乎从零翻文件，而是能先通过 qmd 快速定位相关材料。

### 4. 当“读取前的召回器”

这个姿势也很对。

不要让 agent 一上来就 `read` 一堆大文件，那是 token 自杀。

更好的模式是：

1. 先让 qmd 搜候选
2. 拿到 docid / path / snippet / context
3. 再只读取最相关的少量正文
4. 最后再综合回答或写作

这会让 OpenClaw 的文件消费方式聪明很多。

## 五、qmd 的几个真优势，不是宣传词，是实打实有用

### 1. 本地优先

这是第一大优点。

很多“知识检索”工具，一上来就把你的文档往云上送。

qmd 走的是本地路线：

- 索引本地文件
- 本地 embeddings / rerank
- 通过 node-llama-cpp + GGUF 跑模型

对隐私敏感场景，这非常有吸引力。

尤其你这里已经明确偏好本地、谨慎出网、重视隐私和注入风险，这方向是合拍的。

### 2. 同时兼顾传统搜索和语义搜索

只做关键词，不够灵活；
只做语义，又容易飘。

qmd 把两者结合起来，是比较成熟的思路。

尤其在知识库检索里，这俩经常是互补关系：

- 精确术语、代码符号、报错 → 关键词强
- 模糊概念、会议讨论、自然语言问题 → 语义强

混合检索不是噱头，是真有必要。

### 3. Context 设计挺聪明

仓库里特别强调 `context add` 这一套，意思是：

- 给 collection 或路径树加描述性上下文
- 这些 context 会随着命中的子文档一起返回
- 有助于 LLM 做更好的相关性判断

这点我挺认同。

因为真正让检索变好用的，不只是向量模型多猛，而是：

> **文档本身有没有被放进一个可理解的上下文树里。**

比如同样一篇 `notes.md`：

- 没 context：鬼知道这是啥
- 有 context：知道它属于“项目 A 的架构会议记录”或者“个人实验笔记”

对 agent 决策帮助很大。

### 4. 对 agent 工作流很友好

从 README 看，它明显不是纯给人手敲着玩的。

比如：

- `--json`
- `--files`
- `get --full`
- `multi-get`
- MCP server

这些都说明它在认真考虑“怎么把结果交给另一个系统消费”。

这点就特别适合 OpenClaw 这种编排层。

## 六、qmd 的弱点和坑，也得讲清楚

这部分不说，就是耍流氓。

### 1. 它不是零维护

任何本地索引系统都不是装完就飞升。

你还是得处理：

- collection 怎么划
- pattern 怎么写
- ignore 怎么配
- 什么时候 update
- 什么时候 embed
- 文档切分效果好不好
- context 树是不是乱

如果资料本身组织得稀烂，qmd 也救不了你。

### 2. 模型和硬件成本不是空气

它提到基于 node-llama-cpp 和 GGUF 跑 embedding / rerank / query expansion。

这意味着：

- 首次配置不一定轻松
- 模型文件要占空间
- 本机 CPU / GPU / VRAM 会影响体验
- 长驻 HTTP MCP 虽好，但也更吃资源

所以别把它想成一个零成本小脚本。

它是认真在干活的本地检索系统，不是 bash alias。

### 3. 结果质量高度依赖索引策略

这点非常真实。

结果好不好，不只看模型，还看：

- 文档格式是否统一
- 切分是否合理
- context 是否正确
- collection 是否分得有脑子
- 噪音文档有没有排除

你如果把垃圾堆整堆喂进去，再好的 rerank 也只能在垃圾堆里选相对顺眼的垃圾。

### 4. 不该替代显式记忆治理

这是个很容易犯的错。

有了 qmd，不代表就不用维护 OpenClaw 的 `MEMORY.md` / daily memory / retained knowledge 了。

因为检索层和记忆层本来就不是一个东西。

- qmd 更像“图书馆检索”
- memory 更像“长期认知与偏好”

把两者混成一锅，很容易让系统又啰嗦又不稳定。

### 5. MCP 化以后，边界管理会更重要

一旦以 MCP server 形式常驻，你就得认真考虑：

- 谁能调用
- 索引覆盖哪些目录
- 会不会误扫敏感路径
- 结果里会不会把不该暴露的内容带出来

本地工具不等于天然安全。乱配一样出事。

## 七、qmd 和 OpenClaw memory 的正确关系

这部分很重要，我单独拎出来讲。

### 不要这样理解

错误理解：

> “既然 qmd 能搜文档，那我以后就不需要 memory 了。”

这想法挺蠢的。

因为搜索命中的，是“文档里有什么”；
memory 维护的，是“系统应该稳定记住什么”。

### 更合理的关系是

#### memory 负责：
- 用户偏好
- 长期规则
- 安全边界
- 关键决策
- 稳定、低噪音、可复用事实

#### qmd 负责：
- 原始资料池
- 历史长文
- 会议记录
- 项目文档
- 细节查找与召回

#### OpenClaw 负责：
- 先用 memory 确定行为边界和人设偏好
- 再按任务需要调用 qmd 做素材召回
- 最后综合输出结果

这就顺了。

## 八、对 OpenClaw 来说，最值得用 qmd 的几个场景

### 1. 写研究报告前先搜旧材料

这几乎是最值的用法。

比如要写一篇新文章，OpenClaw 可以先用 qmd 查：

- 之前有没有写过类似主题
- 历史报告里有哪些相关结论
- memory / notes / docs 里有无可复用素材
- 哪些旧文可以互相引用

这样新报告不会每次都像失忆重启。

### 2. 项目知识问答

比如问：

- “我们之前怎么配这个服务的？”
- “某个项目当时为什么选这个方案？”
- “这个错误以前在哪次记录里提过？”

这类问题非常适合 qmd 检索，然后由 OpenClaw 读关键文档、组织答案。

### 3. 会议纪要 / 日报 / 周报检索

会议记录、日报这种东西，人工翻最烦。

qmd 这种支持自然语言检索和 rerank 的系统，就很适合在这类文本堆里捞信息。

### 4. 给 agent 提供候选资料清单

OpenClaw 在启动复杂任务前，可以先让 qmd 输出：

- 相关文件列表
- 相关片段摘要
- 最有希望的几个 docid

然后再决定读哪些文件。

这会比暴力扫目录优雅太多。

### 5. 做“本地知识库版 RAG”

如果以后你想让某些回答更依赖本地文档而不是纯模型脑补，qmd 就很适合做 RAG 的召回层。

OpenClaw 则负责：

- query reformulation
- tool routing
- retrieval 后整合
- 用户可读输出

这组合是成立的。

## 九、最不推荐的几种接法

### 1. 一上来全盘索引整个家目录

这基本属于作死邀请函。

问题包括：

- 噪音巨大
- 命中变脏
- 容易扫进敏感文件
- 维护失控

正确做法：**从明确、干净、主题清晰的目录开始。**

### 2. 把 qmd 当数据库万能替代

它适合文档检索，不是拿来替代结构化数据库、事务系统、权限平台的。

别拿锤子拧螺丝。

### 3. 把 rerank 当真理机

rerank 是排序增强，不是事实担保。

它能帮你把相关材料排前面，但不能代替核验。

### 4. 直接让外部可访问服务暴露整个知识库

尤其如果配成 HTTP MCP，再和别的组件桥接，权限边界一定得想清楚。

本地跑 ≠ 自动安全。

## 十、对 lemon / 可乐 当前系统的具体建议

这里直接说人话，不拐弯。

### 建议一：qmd 非常适合先拿来索引“长期内容资产”和“工作资料”

优先考虑这些目录：

- `projects/lemon-journal-site/src/content/essay`
- `reports/`
- 某些明确的 `notes/` 或项目文档目录
- 经过筛选的会议纪要目录

这样收益大、风险相对低。

### 建议二：不要先碰 MEMORY.md 和隐私高敏目录

哪怕技术上能索引，我也建议先别这么干。

原因很简单：

- MEMORY.md 里是高浓度、精选、长期记忆，本来就不大
- 它更适合通过现有 memory 机制精确读取
- 一旦丢进通用检索，容易让边界变糊

高敏目录就更别说了，纯属给自己埋雷。

### 建议三：先用 CLI 验证效果，再考虑 MCP 常驻

这是最务实的路线：

1. 先建少量 collections
2. 跑 `search / vsearch / query / get`
3. 看召回质量和噪音情况
4. 调整 context 和目录组织
5. 稳了再考虑常驻 HTTP MCP

别一上来就 daemon、HTTP、全系统接入，容易把自己搞成配置民工。

### 建议四：把 qmd 当“召回器”，别当“终裁者”

对 OpenClaw 来说，最好的用法不是：

- qmd 搜到什么就直接回答什么

而是：

- qmd 先给候选
- OpenClaw 再读取关键原文
- 最后按上下文和用户问题组织回答

这样又快又稳。

### 建议五：给 collection 和 context 认真命名

这事看起来小，实际影响很大。

如果 collection 名都叫：

- docs1
- docs2
- misc
- temp

那后面检索质量和可维护性都会烂掉。

更好的方式是按语义命名：

- `journal-essay`
- `project-docs`
- `meeting-notes`
- `reports`
- `workspace-notes`

然后配上清楚的 context。

这是在给未来省命。

## 十一、一个现实可行的接入路线图

### Phase 1：小范围试点

目标：验证 qmd 在你这套环境里到底有没有料。

- 只索引一两个目录
- 跑 CLI
- 测试关键词、语义、混合检索差异
- 调整 context
- 观察结果质量

### Phase 2：形成稳定召回层

目标：让 OpenClaw 在写作、答疑、回顾旧资料时稳定利用它。

- 建立更清晰的 collections
- 形成固定检索流程
- 明确哪些任务先查 qmd
- 让输出以候选文件 + 关键片段为主

### Phase 3：再考虑 MCP 化与长驻服务

目标：降低重复加载成本，提高 agent 调用便利性。

- 配 MCP stdio 或 HTTP
- 控制暴露范围
- 只给需要的 agent / 流程用
- 监控资源占用与维护复杂度

这个顺序非常土，但能少踩很多坑。

## 十二、最后的判断：qmd 不是记忆本身，而是把“本地知识”变得能用

如果只看一句宣传词，很多工具都像神仙。

真正有价值的判断，是看它在系统里承担什么角色。

对 OpenClaw 而言，tobi/qmd 最有价值的定位不是“替代记忆”，也不是“替代数据库”，而是：

> **把散落在本地文件系统里的知识资产，变成一个 agent 能高效调用的召回层。**

这事说起来不花哨，但非常重要。

因为很多代理系统不是死在模型不够聪明，而是死在：

- 找不到资料
- 找资料太慢
- 找到的东西太脏
- 每次都重复读大文件
- 历史内容无法复用

qmd 正好在补这些坑。

所以我的结论很明确：

- **如果你有一批本地文档资产，希望 OpenClaw 更会“翻旧账”，qmd 值得认真试。**
- **如果你还没有整理过资料目录，想靠 qmd 一键拯救混乱，那它救不了你。**
- **如果你重视隐私、本地控制、agent 工作流集成，它和 OpenClaw 的组合方向是对的。**

说白了：

> **qmd 让本地知识更容易被找到，OpenClaw 让这些知识真的开始参与工作。**

前者像图书馆索引，后者像真正会办事的馆员。

一个只会藏书没用；一个只会嘴炮也没用。

俩接上，才像回事。
